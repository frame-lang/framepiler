# RFC-0015: Factory-Only Construction + `@@!` Blank Allocation

**Author:** Mark Truluck <mark@frame-lang.org>
**Status:** Shipped (factory-only + `@@[create]`/`@@[save]`/`@@[load]`, framec 4.1.0+). `@@!Foo()` blank-allocation syntax is **proposed for 4.2.x** — see Implementation status.
**Created:** 2026-05-03
**Last revised:** 2026-05-10
**Builds on:** RFC-0012 (`@@[persist]`), RFC-0013 (annotation syntax), RFC-0014 (`@@[main]`)

---

## Summary

Frame separates **construction** (initialization with parameters) from **allocation** (creating an empty instance). Three system-level lifecycle attributes name the three operations users invoke:

- `@@[create(<name>)]` — the factory: allocates + initializes via the start state's `$Start(...)` body and `$>` handler
- `@@[save(<name>)]` — serializes the system's persistent state to bytes
- `@@[load(<name>)]` — deserializes bytes back into the system, **bypassing all init code**

The Frame source-level expression for "give me an empty instance with no init code run" is the new sigil **`@@!Foo()`**. Combined with `@@[load]`, it provides clean two-line restore:

```frame
fresh = @@!Foo()           // allocates without calling init
fresh.restore_state(data)   // overwrites domain + compartment from saved bytes
```

This RFC supersedes RFC-0012's operation-attribute persist form. The hard cut at framec 4.1.0 is enforced by validator code **E819**.

## Motivation

### The original problem

Two real-world bugs surfaced from the *Frame Arcade* and *CCA* projects exposed the same architectural seam:

1. **`@@:return(self._inner(args))` drops a paren on nested calls** — pure codegen bug, fixed in GDScript independently of this RFC.
2. **Parameterized sub-system restore calls `Sub.new()` with zero args and crashes when init requires parameters** — this is *architectural*. It surfaces the question: how should framec reconstruct a nested `@@system` instance during persist restore when that system has required init parameters?

The second question forks into 17 language-by-language nuisances: GDScript's `_init` cannot be skipped, Java's reflection trickery to instantiate without a public no-arg constructor, Swift's `required init`, Dart's factory vs. unnamed constructor, Erlang's `gen_statem:init/1`, etc.

The original framec design put *user code* in each backend's natural constructor. That choice made every backend answer "how do I allocate without running user code?" — a question with 17 different answers, two of which are unreliable, one of which (GDScript) had no clean answer at all.

### The fix

**Frame stops emitting user code in host-language constructors entirely.** The host constructor only allocates. Construction with user intent (parameters, derived state) is a state-machine handler — specifically the start state's `$Start(...)` body and `$>` handler. Restore becomes "allocate blank, then deserialize bytes into the blank instance" — uniform across all 17 backends.

Issue #2 stops being a defect class because there is no longer a user-defined constructor that fails on zero-arg calls.

## The contract

### Construction

A `@@system` declaration's three-group header — `$(state-args)`, `$>(enter-args)`, `bare-domain-args` — defines the construction surface. Framec emits an auto-generated **factory** whose signature mirrors the header. The factory body is mechanical:

1. Allocate a blank instance via the framec-internal blank allocator
2. Route each argument per its sigil (state-args → compartment, enter-args → compartment, bare → domain initializers)
3. Fire the start state's `$Start(...)` body, then its `$>` handler
4. Return the fully-initialized instance

User code never lives in the factory body. Init logic — derived field computation, validation, resource acquisition — belongs in `$Start(...)` and `$>`. This is a hard rule.

### Save and load

`@@[save(<name>)]` and `@@[load(<name>)]` declare the user-facing names of the serialization operations. The format is dictated by the system-level `@@[persist(<Format>)]` attribute (e.g., JSON). Framec generates both bodies.

- **Save** is an instance method: `bytes = inst.save_state()`. It serializes domain fields + compartment + state stack + any nested `@@system` domain fields.
- **Load** is an instance method: `inst.restore_state(bytes)`. It deserializes the bytes and overwrites the instance's domain + compartment **without firing any init code**.

### Blank allocation via `@@!Foo()`

To use `@@[load]` cleanly without re-running init, the user needs a way to allocate a blank instance. Frame source expresses this with the new sigil:

```frame
fresh = @@!Foo()
```

`@@!Foo()` is **always zero-arg** and **doesn't call init**. Per-backend rendering uses each language's natural blank-allocation primitive (`object.__new__` in Python, `Object.create(Foo.prototype)` in JS, `Ruby Foo.allocate`, etc. — full table below).

The standard restore pattern:

```frame
data = inst.save_state()       // earlier: serialize
                                // ... later, possibly different process ...
fresh = @@!Foo()               // allocate blank
fresh.restore_state(data)      // overwrite from saved bytes
```

This is symmetric with the framec-internal composition restore path (D6) — the user-facing `@@!Foo()` exposes exactly the mechanism nested-system restore uses internally.

### Where init logic lives

Always in the start state's `$>` handler (and optionally the `$Start(...)` body):

```frame
@@system Inner($>(seed: int)) {
    machine:
        $Initializing {
            $>(seed: int) {
                self.derived = compute(seed)
                self.connection = open_connection(seed)
            }
            ready() { -> $Active }
        }
        $Active {}
    domain:
        derived: int = 0
        connection: Connection = null
}
```

The factory (`@@Inner($>(7))`) calls init. `@@!Inner()` doesn't. That's the entire architectural point.

### Composition

When a system declares another `@@system` as a domain field:

```frame
@@system Outer {
    domain:
        inner = @@Inner(42)
}
```

Construction calls `Inner`'s factory with `42`. Restore — both `Outer.restore_state(...)` and any nested-system restore inside it — uses `@@!Inner()` (the blank allocator) followed by recursive load. The Issue #2 reproducer stops crashing because there is no longer a user-defined `_init(seed: int)` that demands an arg — the blank allocator is zero-arg by definition.

### Post-load resource reattach

Fields tagged `@@[no_persist]` are excluded from save and load. After restore they hold their `domain:` default (typically `null` for resource handles). Reattach is a user concern — Frame does not provide an implicit hook.

**Pattern A — explicit reattach operation** (recommended for most cases):

```frame
@@system Worker {
    interface:
        do_work()
    operations:
        reattach()
    machine:
        $Working {
            do_work() { ... }
        }
    domain:
        last_seq: int = 0
        @@[no_persist]
        connection: Connection = null
}
```

User flow:

```frame
fresh = @@!Worker()           // blank
fresh.restore_state(data)      // domain restored; connection still null
fresh.reattach()              // user op opens connection
fresh.do_work()                // ready
```

**Pattern B — state-machine driven reattach.** Some users prefer to model the reattach as an event that drives a transition through a "reconnecting" state. Both patterns are pure Frame; framec doesn't need to support either specifically.

## Lifecycle attribute reference

The four attributes — three on the system header, one as a Frame-source sigil — collectively express system lifecycle.

### Attribute summary

| Form | Position | Argument | Purpose |
|---|---|---|---|
| `@@[create(<name>)]` | system-level attribute | optional identifier | rename the user-facing factory |
| `@@[save(<name>)]` | system-level attribute | optional identifier | rename the save op |
| `@@[load(<name>)]` | system-level attribute | optional identifier | rename the load op |
| `@@!Foo()` | Frame-source expression | none (zero-arg only) | allocate a blank instance |

The first three are identifier-name overrides; the fourth is a syntactic primitive. The user has no naming control over `@@!Foo()` because there is no host-language name to expose — it always renders to the per-backend blank-allocation primitive.

### Per-backend rendering — factory + blank allocation

| Backend | Factory call (e.g. `@@Foo(7)`) | Blank allocation (`@@!Foo()`) |
|---|---|---|
| Python | `Foo(7)` (delegating `__init__`) | `Foo.__new__(Foo)` |
| JavaScript | `new Foo(7)` | `Object.create(Foo.prototype)` |
| TypeScript | `new Foo(7)` | `Object.create(Foo.prototype)` |
| Java | `new Foo(7)` | synthesized package-private `Foo.__blank()` |
| Kotlin | `Foo(7)` | synthesized `Foo.__blank()` companion fn |
| C# | `new Foo(7)` | synthesized internal `Foo.__blank()` static |
| Rust | `Foo::new(7)` | synthesized `Foo::__blank()` (uses `MaybeUninit` or all-fields-Default) |
| Swift | `Foo(7)` | synthesized `Foo.__blank()` static func |
| GDScript | `Foo.create(7)` | `Foo.new()` (the empty `_init` IS the blank allocator) |
| Go | `NewFoo(7)` | `&Foo{}` zero literal |
| C | `Foo_create(7)` | `Foo_alloc()` (calloc + zero-init) |
| C++ | `Foo::create(7)` | `Foo()` (default ctor, empty) |
| Dart | `Foo.create(7)` | `Foo.__blank()` synthesized private named ctor |
| PHP | `new Foo(7)` | `(new ReflectionClass(Foo::class))->newInstanceWithoutConstructor()` |
| Ruby | `Foo.new(7)` | `Foo.allocate` |
| Lua | `Foo.create(7)` | `setmetatable({}, Foo)` |
| Erlang | `foo:start_link(7)` | `foo:start_link_blank()` (synthesized clause with empty Data) |

Defaults for save/load names (when `@@[save(<name>)]` / `@@[load(<name>)]` are absent):

| Backend | Default save name | Default load name |
|---|---|---|
| Python, GDScript, Lua, Erlang | `save_state` | `restore_state` |
| Rust, C++ | `save_state` | `restore_state` |
| C | `Foo_save_state` | `Foo_restore_state` |
| Java, Kotlin, Swift, Dart, PHP | `saveState` | `restoreState` |
| C#, Go | `SaveState` | `RestoreState` |
| JavaScript, TypeScript, Ruby | `save_state` | `restore_state` |

## Validator rules

| Code | Trigger | Message |
|---|---|---|
| **E815** | `@@[create]`, `@@[save]`, or `@@[load]` attached to anything other than a system header | "lifecycle attribute must appear at system level, not at \<position\>" |
| **E817** | name argument is not a valid host-language identifier | "lifecycle attribute name '\<x\>' is not a valid identifier in target \<lang\>" |
| **E818** | duplicate `@@[save]` or `@@[load]` on the same system | "duplicate lifecycle attribute on system \<Foo\>" |
| **E819** | RFC-0012 op-attribute persist form (`operations: @@[save] foo(): T`) | hard-cut migration message + codemod pointer |
| **E820** | `@@!Foo(args)` with non-empty args | "blank allocation `@@!Foo()` is zero-arg by definition; received: \<args\>" |
| **E821** | `@@!Foo()` references a system not declared in the module | "undefined system '\<Foo\>' in blank allocation" |

E820 + E821 are introduced by the `@@!` proposal; the rest are already shipping.

## Migration

### From framec 4.0.x (RFC-0012 form) to 4.1.x

Hard-cut at 4.1.0. The old operation-attribute form is rejected by **E819**. A codemod ships at `scripts/migrate_rfc0015.py` that:

1. Lifts `operations: @@[save] X(): T` → system-level `@@[save(X)]` above `@@system`.
2. Removes the lifted operation declarations from `operations:`.
3. If `operations:` becomes empty, removes the block.
4. Is AST-based (not regex), idempotent, and was used to migrate the ~4,832-fixture corpus.

### From framec 4.1.x (current) to 4.2.x (proposed `@@!Foo()`)

Source-additive. Existing user code is unaffected. The new `@@!Foo()` syntax is a new affordance, not a replacement. Users who currently work around the awkward restore by constructing with placeholder args:

```frame
// 4.1.x — awkward but works
fresh = @@Inner($>(0))         // factory fires $> with 0 (wasted work)
fresh.restore_state(data)      // overwrite

// 4.2.x — clean
fresh = @@!Inner()             // blank, no $> runs
fresh.restore_state(data)      // overwrite
```

can opt in by switching the construction line. No breaking change.

## Locked decisions

These decisions are binding for their respective releases. Revisiting requires a new RFC.

### D1 — Hard-cut, not additive parser (4.1.0, locked 2026-05-03)

RFC-0012's operation-attribute form is rejected by framec 4.1.0+ via E819. Carrying two parsing forms forward muddles the AST and documentation; the codemod handles existing corpus migration.

### D2 — Locked default factory/save/load names per backend (4.1.0)

See the per-backend rendering table above. These are baked into per-language guides.

### D3 — Erlang fit (resolved 2026-05-03 via spike)

The Erlang spike produced `frame_restore_mode` flag in the Data record. Every state's enter clause checks the flag; load_state uses tagged init args. ~50–80 LOC.

### D4 — Restore must skip start-state init logic (4.1.0)

`load(data)` does NOT call init. Test 96 verifies this across all 17 backends. Result: every backend already upholds D4 cleanly without additional codegen changes — pickle/Jackson/serde/etc. handle the bypass internally.

### D5 — Zero-arg factory case (4.1.0)

Systems with no `@@system`-level params auto-generate a zero-arg factory. Same shapes as D2 with empty args.

### D6 — Composition restore (4.1.0)

Nested-system restore uses the framec-internal blank allocator + recursive load, NOT the user factory. This is the explicit fix for FRAMEC_BUGS.md Issue #2.

### D7 — `@@!Foo()` blank-allocation syntax (4.2.x — proposed, this revision)

The user-facing exposure of the framec-internal blank allocator. Always zero-arg; rendered per-backend; never fires init code.

**Why a sigil and not a fourth attribute:** the other three lifecycle attributes (create, save, load) accept a name argument because they map to host-language method names users invoke. `@@!Foo()` has no host-language name to expose — the call site is Frame source. A fourth `@@[blank(<name>)]` attribute would offer a knob with no purpose. The `@@!` digraph is unambiguous in Frame's grammar (currently `!` is not a Frame sigil) and visually distinguishes "this skips init" with one extra character.

**Signature:** zero-arg only. `@@!Foo(anything)` is rejected by **E820**. Args have nowhere to go since the entire point is "no factory body runs."

**Position:** any expression position where `@@Foo(args)` is legal.

**Rendering:** per the per-backend table above. For backends without a clean primitive (Java, Kotlin, Swift, Rust, Dart), framec synthesizes a `__blank()` companion / static / private named ctor that performs the equivalent.

## Rejected designs

### `@@[destroy]`

Destruction semantics are non-uniform across hosts (6 of 17 backends have reliable destructor hooks; 11 don't). Frame's value proposition is determinism. Users who need teardown logic write it as a regular event handler or operation and invoke it explicitly. The state-machine model is sufficient.

### `@@[blank(<name>)]` attribute (rejected during 2026-05-10 design)

Considered as a fourth lifecycle attribute mirroring create/save/load. Rejected because there is no host-language name for the user to choose: `@@!Foo()` is the call site, no `inst.somename()` is invoked. An attribute would be a knob without a purpose. The Frame-source sigil `@@!` is the right shape.

### Static `<load_op>` returning `Self` (rejected during 2026-05-10 design)

Considered as an alternative to `@@!`: make load a class/static method that returns a freshly-restored instance, hiding the blank-allocation step:

```
fresh = Inner.unpickle(data)   // hypothetical static
```

Rejected because:
- Breaks the existing instance-method `restore_state(self, data)` pattern that ships in 4.1.x
- Hides the two-step nature in a way that prevents users from doing other things with the blank instance (e.g., setting up a non-persisted dependency before load)
- More codegen surface (must change save/load shape across all 17 backends)

`@@!Foo()` exposes the blank instance as a first-class value; users can do whatever they want with it, including but not limited to calling load.

## Implementation status

### Shipped in framec 4.1.0+ (2026-05-03 → 2026-05-04)

- Phase 0 lockdown: D1–D6 decisions locked
- Phase 1.0 Erlang spike: `frame_restore_mode` flag design verified
- Phase 1.1a–1.9: `@@[create(<name>)]` factory rename works on all 17 backends
- Phase 1.1d save/load: rename works on all 17 backends
- Phase 5 D4 verification: Test 96 across all 17 backends, all pass
- Phase 6 codemod: `scripts/migrate_rfc0015.py` migrates ~4,832 fixtures
- Validators E815, E817, E818, E819 in place
- 4,781/4,781 matrix tests pass
- `@@[on_load]` post-restore hook removed (-150 LOC of validator + codegen)

### Proposed for 4.2.x — D7 (`@@!Foo()` blank allocation)

Not yet implemented. The implementation plan is at `_scratch/at_bang_implementation_plan.md` (gitignored). High-level phasing:

1. **Phase A** — Python-only spike (~2h)
2. **Phase B** — Easy backends rollout: Ruby, JS, TS, Lua, Dart, GDScript, C, Erlang (~half day)
3. **Phase C** — Medium backends: C++, Go, PHP (~half day)
4. **Phase D** — Hard backends with synthesized `__blank()`: Rust, Java, C#, Kotlin, Swift (~1–2 days)
5. **Phase E** — Validator codes E820, E821 (~1h)
6. **Phase F** — Documentation (RFC, language ref, 17 per-language guides, cookbook recipe) (~half day)
7. **Phase G** — Matrix test for restore-via-`@@!` round-trip across 17 backends (~half day)

Total estimate: 3–5 working days. Comparable in scope to the original RFC-0015 Phase 1.x rollout.
