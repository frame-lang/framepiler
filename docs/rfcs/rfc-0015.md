# RFC-0015: `@@[create]` — Factory-Only Construction

**Prompt Engineer:** Mark Truluck <mark@frame-lang.org>
**Status:** Draft (2026-05-03)
**Created:** 2026-05-03
**Builds on:** RFC-0012 (`@@[persist]` + `@@[save]`/`@@[load]`), RFC-0013
(annotation syntax), RFC-0014 (`@@[main]`)

---

## Motivation

Two real-world bugs surfaced from the *Frame Arcade* and *CCA* projects
expose the same architectural seam:

1. **`@@:return(self._inner(args))` drops a paren on nested calls** —
   pure codegen bug, not architectural; fixable in the GDScript emitter
   independently of this RFC.
2. **Parameterized sub-system restore calls `Sub.new()` with zero args
   and crashes when `_init` requires parameters** — this *is*
   architectural. It surfaces the question: how should framec
   reconstruct a nested `@@system` instance during persist restore when
   that system has required `_init` parameters?

The second question forks into a series of language-by-language
nuisances: GDScript's `_init` cannot be skipped, Java's reflection
trickery to instantiate without a public no-arg constructor, Swift's
`required init`, Dart's factory vs. unnamed constructor, etc. Today,
framec puts *user code* in each backend's natural constructor (`_init`
on GDScript, `__init__` on Python, public ctor on Java/C#, `pub fn new`
on Rust). That choice means every backend has to answer "how do I
allocate without running user code?" — a question that has 17 different
answers, two of which are unreliable, one of which (GDScript) has no
answer at all.

This RFC removes the question by removing the premise. **Frame stops
emitting user code in host-language constructors entirely.** The only
thing the host constructor does is allocate. Construction with user
intent (parameters, derived state) becomes a state-machine handler —
specifically `$>` of the start state — which already exists and already
runs once at construction. A `@@[create]` attribute provides a
host-facing name escape hatch for the auto-generated factory, but it
takes no body.

Restore becomes trivial: framec calls the empty allocator, then
`@@[load]`. No user code runs on restore. Issue #2 stops existing as a
defect class.

---

## Design

### The factory-only contract

For every `@@system`, framec emits exactly two construction-side
artefacts:

1. **A blank allocator.** Always emitted. Always zero-arg. Always
   empty-body. The host-language native equivalent of "allocate an
   instance with all fields at their type-default." Per-backend:

   | Backend | Blank allocator |
   |---|---|
   | Python | empty `def __init__(self): pass` |
   | JS / TS | empty `constructor() {}` |
   | Java / C# / Kotlin | empty package-private no-arg ctor |
   | Swift | empty `init()` |
   | GDScript | empty zero-arg `func _init():` |
   | Rust | `Default::default()` (or a derived `#[derive(Default)]`) |
   | C | `Inner_alloc()` (calloc + zero-init) |
   | C++ | empty default ctor |
   | Go | `&Inner{}` zero literal |
   | Dart | private named ctor `Inner._blank()` |
   | PHP | empty `__construct()` |
   | Ruby | empty `initialize` |
   | Lua | empty table init |
   | Erlang | `init/1` returning blank `Data` |

2. **An auto-generated factory.** Always emitted. Args match the
   system's declared params. Body is purely positional: assign each
   param into the corresponding domain field. **No user code.**

   Default name is `_create` (private/internal); the host-natural
   name (per backend) is what users write at the call site. Default
   mappings:

   | Backend | Default factory call site |
   |---|---|
   | Java / C# / Kotlin | `new Inner(seed)` (public ctor) |
   | C++ | `Inner::create(seed)` static method |
   | Rust | `Inner::new(seed)` static method |
   | Python | `Inner(seed)` (`__init__` delegates to `_create`) |
   | JS / TS | `new Inner(seed)` (`constructor` delegates) |
   | GDScript | `Inner.create(seed)` static method |
   | Swift | `Inner(seed)` (delegating init) |
   | Go | `NewInner(seed)` package function |
   | Dart | `Inner.create(seed)` factory ctor |
   | PHP | `new Inner($seed)` (`__construct` delegates) |
   | Ruby | `Inner.new(seed)` (overridden allocator) |
   | Lua | `Inner.create(seed)` |
   | C | `Inner_create(seed)` |
   | Erlang | `inner:start_link(seed)` |

   The Frame source always writes `@@Inner(seed)`; per-backend
   rendering picks the right host idiom.

The blank allocator is **framec-internal**. User code never sees it
directly. Restore uses it.

### `@@[create(<name>)]` — system-level attribute, name-only

The `@@[create]` attribute is a **system-level attribute** that takes
a single optional argument: the desired user-facing factory name.
The signature is **always** derived from the `@@system` header's
three-group parameter declaration; the user has no freedom over
shape, only over the rendered name.

Per `frame_language.md` § "System Parameters", every system header
declares its construction surface in three ordered, optional groups:

| Sigil | Routes to |
|---|---|
| `$(name: type)` | start state's `compartment.state_args` |
| `$>(name: type)` | start state's `compartment.enter_args` |
| `name: type` (bare) | **domain param** — in scope for `domain:` field initializers |

The complete lifecycle declaration lives entirely at the system
header:

```frame
@@[target("python_3")]
@@[persist(JSON)]                   // declares format → save/load signatures
@@[create(make)]                    // optional: rename factory
@@[save(pickle)]                    // optional: rename save op
@@[load(unpickle)]                  // optional: rename load op
@@system Inner($(x: int), $>(mode: str), name: str) {
    machine:
        $Start(x: int) {
            $>(mode: str) {
                self.derived = compute(x, mode)
            }
            describe(): str { @@:(self.name + "@" + str(x)) }
        }
    domain:
        name = name
        derived: int = 0
}
```

User-facing call site (Frame source):

```frame
inst = @@Inner($(7), $>("fast"), "R2D2")    // factory-named per @@[create(make)]
data = inst.pickle()                          // save op named per @@[save(pickle)]
fresh = @@Inner($(0), $>(""), "")            // blank instance
fresh.unpickle(data)                          // load op named per @@[load(unpickle)]
```

The four lifecycle attributes are all **system-level**, all
**name-only**. Operations: block is no longer used for lifecycle —
it's reserved for user-defined operations (Frame syntax for "method
that operates on the system but isn't an event handler"). This is
a clarifying narrowing of `operations:`'s purpose.

The user picks:

- **Names**, expressed as the single string argument to each
  lifecycle attribute. Defaults per-backend if absent.
- *Nothing else.* All signatures are fully derived from `@@system`
  header + `@@[persist(<Format>)]`.

Framec generates everything else:

- **`@@[create]` body**:
  1. Allocate a blank instance via the framec-internal allocator.
  2. Populate `compartment.state_args` from each `$(...)` arg.
  3. Populate `compartment.enter_args` from each `$>(...)` arg.
  4. Bring bare args into scope and run domain field initializers.
  5. Fire the start state's transition (dispatches `$Start(...)`
     body and `$>(...)` handler in that order).
- **`@@[save]` body**: serializer for the chosen format that walks
  every domain field, the compartment, and the state stack.
- **`@@[load]` body**: deserializer for the chosen format that
  overwrites compartment + domain in place.
- **Host-language rendering**. Each lifecycle op is rendered into
  natural host syntax per backend.

### Why everything must be framework-generated

Construction in the factory-only model has nothing for the user to
write. The body is mechanical: route each arg per its sigil, then
fire the start-state transition. There's only one correct
implementation. Allowing a user body would re-introduce the problem
this RFC is solving — user code running on a path framec needs to
also use for restore.

User init logic — derived fields, validation, resource acquisition —
belongs in the start state's `$Start(...)` body and `$>(...)`
handler. Those are the hooks Frame already provides. They run on
construction (because constructing means firing the initial
transition into the start state) but **not** on restore (which
restores the compartment directly via the framework-generated
`@@[load]`, bypassing construction).

The same logic applies to save and load: the format is the user
choice; the body is mechanical given that format.

### Default emission when lifecycle attributes are absent

For each of `@@[create]`, `@@[save]`, `@@[load]`:

- If the attribute is absent (or written without a name argument),
  framec uses a per-backend default: `Inner.create` / `inst.save` /
  `inst.load`, rendered idiomatically (public ctor on Java/C#, static
  method on GDScript, etc.).
- If the attribute provides a name, framec uses that name.

For trivial zero-param systems, the create factory has a zero-arg
signature; nothing else changes.

### Where init logic lives

User init logic — validation, derived-field computation, external
resource acquisition — belongs in the **start state's `$>` handler**:

```frame
@@system Inner(seed: int) {
    domain:
        seed: int = seed
        derived: int = 0
        connection: Connection = null

    machine:
        $Initializing {
            $>() {
                self.derived = compute(self.seed)
                self.connection = open_connection(self.seed)
            }
            ready() { -> $Active }
        }
        $Active {}
}
```

This is the natural Frame model. State entry is when work-on-state-entry
runs. The start state's entry is when work-on-system-creation runs.
Frame already does this; this RFC just commits to it as the *only*
place where construction-time logic lives.

### Restore semantics

Restore is now mechanical and uniform across all 17 backends:

1. Framec calls the blank allocator (zero-arg empty).
2. Framec calls `@@[load]` with the saved bytes.
3. `@@[load]` writes domain fields and reconstructs the compartment.
4. **Done.** No user code from `@@[create]` runs. No `$>` handler
   fires. The restored instance is byte-equivalent to the saved one
   for the persisted-fields subset.

The restored instance has whatever derived state the original had at
save time (because that state was in the saved bytes). External,
non-persistable resources (TCP connections, file handles, GPU
contexts, etc.) are *not* restored — they're not part of persist
scope. They have whatever value their `domain:` default specifies
(typically `null`).

### Post-load reattach for non-persisted fields

Fields tagged `@@[no_persist]` (RFC-0012) are excluded from save and
load. After restore, they hold their `domain:` default — usually
`null` for resource handles. The user is responsible for
re-establishing them.

Frame doesn't model this with a lifecycle hook. The user uses what
the language already provides:

**Pattern A — explicit reattach operation.** The user declares a
regular operation in `operations:` and invokes it after `@@[load]`:

```frame
@@[persist(JSON)]
@@[load(unpickle)]
@@system Worker {
    interface:
        do_work()

    operations:
        reattach()                        // user-defined, user-invoked

    machine:
        $Working {
            do_work() { ... }
        }

    domain:
        last_seq: int = 0                // persisted
        @@[no_persist]
        connection: Connection = null    // not persisted; null after load
        @@[no_persist]
        cache: Dict = {}                 // not persisted; empty after load
}
```

User flow:

```
worker = @@Worker()                       // create + acquire resources via $> of start
worker.do_work()
data = worker.pickle()
                                          // ... later ...
fresh = @@Worker()                        // blank; resources null
fresh.unpickle(data)                      // domain restored; non-persisted fields stay null
fresh.reattach()                          // user op opens connection, warms cache
fresh.do_work()                           // ready
```

**Pattern B — state-machine driven reattach.** Some users prefer to
model the reattach as an event that drives a transition through a
"reconnecting" state. Both patterns are pure Frame; framec doesn't
need to support either specifically.

This mirrors the same logic that rejected `@@[destroy]`: lifecycle
work in Frame is modeled via state-machine events and user
operations, not via implicit hooks. Resource lifecycle is the
user's domain; framec's contract is to restore the persistable
state byte-equivalent and let the user wire the rest.

### Composition

When a system declares another `@@system` as a domain field:

```frame
@@system Outer {
    domain:
        inner = @@Inner(42)
}
```

Framec emits the composition the same way it emits any other domain
default: it calls `Inner`'s factory with `42`. Restore of `Outer`
calls `Inner`'s blank allocator and then recursively `@@[load]`s
`inner` from the saved nested bytes. The Issue #2 reproducer stops
crashing because there is no longer a user-defined `_init(seed: int)`
that demands an arg — the host constructor is empty.

## Migration

### Source compatibility

- Existing `@@system Inner(seed: int)` declarations continue to work.
  Framec auto-generates the factory body from the system header,
  same as today.
- Existing fixtures that put init code in `_init`-equivalent places
  (which was never explicit Frame syntax — it was emergent behavior
  in some backends) now need that code moved to the start state's
  `$Start(...)` body or `$>(...)` handler. **In practice, the test
  corpus does not do this** — Frame source already writes init logic
  in those hooks because that's the documented pattern. Migration
  is mechanical for any fixtures that drifted.
- The cookbook's 110 recipes were audited 2026-05-03; none rely on
  user code in framec-emitted constructors. Migration cost: zero.
- The 4,832 matrix fixtures were audited at the same time; the
  generators emit only `domain` defaults and `$>` init. Migration
  cost: zero.

### RFC-0012 supersession — hard-cut at 4.1.0

Per **D1 (locked decision above)**, this RFC supersedes RFC-0012's
operation-attribute form for save and load. The 4.1.0 framec
release rejects the old form with **E819** and a migration message.

Old form (RFC-0012, accepted by framec 4.0.x):

```frame
operations:
    @@[save] save_state(): JSON
    @@[load] restore_state(data: JSON)
```

New form (RFC-0015, required by framec 4.1.0+):

```frame
@@[persist(JSON)]
@@[save(save_state)]
@@[load(restore_state)]
@@system Foo { ... }
```

#### Codemod

A Python codemod ships with the 4.1.0 release at
`framepiler/scripts/migrate_rfc0015.py`. It scans `.f*` source
files and:

1. For each `@@[persist]` system with `operations:` block
   containing `@@[save] X(): T` and/or `@@[load] Y(data: T)`:
   - Lifts each name to a system-level `@@[save(X)]` /
     `@@[load(Y)]` attribute, inserted just before the
     `@@system` keyword.
   - Removes the lifted operation declarations from
     `operations:`.
2. If `operations:` is now empty, removes the block entirely.
3. Leaves systems without persist untouched.
4. Leaves `@@system Inner(seed: int)` parameter lists untouched.
5. Idempotent — running it twice on a file produces the same
   output as running it once.

Verification: codemod is AST-based (uses framec's parser, not
regex). On a 50-fixture sample, output is verified manually.
Then run on the full ~4,832-fixture corpus.

#### Test corpus impact

- **Fuzz tests** (~26,565 cases across 18 phases): generator-driven.
  Updating ~3-4 generators (those that emit `@@[persist]` systems)
  to emit the new system-level form is one-line template edits.
  Regenerate; done. Existing `cases_*` directories are gitignored;
  next regen produces the new shape.
- **Hand-written corpus** (~4,832 fixtures): codemod runs once.
  Mechanical, AST-based, no manual editing.

Validators E811/E812/E813 (RFC-0012's signature/no-body checks on
operation-attribute save/load) are removed. New validators E815-E819
(this RFC) replace them with the system-level shape.

### Backend codegen

Per-backend changes:

1. Stop emitting user code in `_init` / `__init__` / public ctor /
   `pub fn new` etc. Instead, emit the empty blank allocator +
   the auto-generated factory whose name comes from
   `@@[create(<name>)]` (or per-backend default).
2. Wire restore to call the empty allocator + `@@[load]`. (Several
   backends already do this for typed-persist via reflection; this
   RFC formalizes the pattern.)
3. The Issue #2 nested-system restore path stops emitting
   `Inner.new()` (zero-arg call to user `_init` that fails); it now
   calls the empty allocator (which is zero-arg by construction).
4. Move save/load codegen entry point from operation-attribute
   resolution to system-level attribute resolution.

The cargo `framec` 4.0.0 release path: this RFC ships as 4.1.0 with
RFC-0012's persist contract (now in its name-only form), RFC-0013's
annotation syntax, and RFC-0014's `@@[main]` attribute as a
coherent module-level lifecycle declaration package.

### Lifecycle attribute adoption

All four optional. The defaults per backend are idiomatic enough
that most users won't need any of them. The attributes exist for
naming control — e.g., a Java user who wants a static factory
`Inner.spawn(seed)` instead of `new Inner(seed)`, or a GDScript
user who prefers `Inner.unpickle(data)` instead of `restore_state`.

## Why this is right

1. **Eliminates a bug class.** Issue #2 (parameterized sub-system
   restore zero-arg crash) stops existing because there is no
   user-defined `_init` to fail. The contract is uniform and
   framec-controlled.

2. **Aligns Frame with the broader language landscape.** Languages
   without implicit constructors (Zig, Rust, Go, C, Lisp records,
   Clojure, Haskell, OCaml, Smalltalk via `basicNew`, HDLs,
   Solidity structs) become natural Frame backends. The current
   model would require framec to *invent* constructor semantics
   that don't exist in those languages. Factory-only doesn't.

3. **Single shape across all 17 backends.** Today framec deals with
   17 different constructor stories. After this RFC, framec deals
   with one — empty allocator + auto-generated factory — rendered
   per-backend. The complexity collapses.

4. **Composes with `@@[save]`/`@@[load]`.** RFC-0012 already
   established that lifecycle attributes mark user-supplied
   operations. `@@[create]` joins the family but with a sharper
   semantic: rename-only, no body. The contract is "the four
   lifecycle attributes (`create`, `save`, `load`) all take
   user-overridable defaults; `create` is rename-only, `save`/`load`
   accept user bodies."

5. **Restore is trivial.** Two-step: allocate-blank, call-load.
   No language-specific "skip init" gymnastics. No reflection
   tricks. No friend-classes. No private constructors with
   special bypass mechanisms.

7. **Symmetric, name-only lifecycle family.** Three system-level
   attributes, one optional argument each, all signatures derived
   from elsewhere in the system header.

   | Attribute | Signature dictated by | User picks |
   |---|---|---|
   | `@@[create(<name>)]` | `@@system` three-group header | factory name |
   | `@@[save(<name>)]` | `@@[persist(<Format>)]` (returns `Format`) | save op name |
   | `@@[load(<name>)]` | `@@[persist(<Format>)]` (takes `Format`) | load op name |

   The user's only freedom is naming. Operations: block is reserved
   for genuine user-defined operations, not lifecycle hooks.

   This supersedes RFC-0012's operation-attribute form for save/load
   (see Migration below).

6. **GDScript stops being special.** The current model makes
   GDScript awkward (its `_init` cannot be bypassed). The new
   model makes GDScript *normal*: the empty `_init()` is the blank
   allocator, exactly like every other backend.

## Locked decisions (Phase 0 — 2026-05-03)

These are the decisions made before any framec implementation
work. Each is binding for the 4.1.0 release; revisiting requires a
new RFC.

### D1 — Hard-cut, not additive parser

**Decision:** RFC-0012's operation-attribute form (`operations:
@@[save] save_state(): JSON`) is rejected by framec 4.1.0. Old
form raises **E819** with a clear migration message. A codemod
ships with the release to migrate hand-written corpus.

**Why:** This is the last big change before release. Carrying two
parsing forms forward leaves the AST and documentation muddled —
every example has to choose one form. Hard-cut means one canonical
shape across the cookbook, language ref, and per-language guides.
RFC-0012's B3 already proved the codemod path works (~425
fixtures); doing it again for ~4,832 is mechanical, not creative.

### D2 — Locked default factory/save/load names per backend

**Decision:** Each backend has a single canonical default for each
of the three lifecycle attributes when no name is supplied. These
become the documented per-language idiom and are baked into the
per-language guides.

| Backend | Default `@@[create]` rendering | Default save | Default load |
|---|---|---|---|
| Python | `Inner(seed)` (delegating `__init__`) | `save_state` | `restore_state` |
| JavaScript | `new Inner(seed)` (delegating `constructor`) | `save_state` | `restore_state` |
| TypeScript | `new Inner(seed)` (delegating `constructor`) | `save_state` | `restore_state` |
| Java | `new Inner(seed)` (public ctor) | `saveState` | `restoreState` |
| Kotlin | `Inner(seed)` (public primary ctor) | `saveState` | `restoreState` |
| C# | `new Inner(seed)` (public ctor) | `SaveState` | `RestoreState` |
| Rust | `Inner::new(seed)` (associated fn) | `save_state` | `restore_state` |
| Swift | `Inner(seed)` (delegating `init`) | `saveState` | `restoreState` |
| GDScript | `Inner.create(seed)` (static; `_init` empty) | `save_state` | `restore_state` |
| Go | `NewInner(seed)` (package fn) | `SaveState` | `RestoreState` |
| C | `Inner_create(seed)` | `Inner_save_state` | `Inner_restore_state` |
| C++ | `Inner::create(seed)` (static) | `save_state` | `restore_state` |
| Dart | `Inner.create(seed)` (factory ctor) | `saveState` | `restoreState` |
| PHP | `new Inner($seed)` (delegating `__construct`) | `saveState` | `restoreState` |
| Ruby | `Inner.new(seed)` (overridden allocator) | `save_state` | `restore_state` |
| Lua | `Inner.create(seed)` | `save_state` | `restore_state` |
| Erlang | `inner:start_link(Seed)` | `save_state` | `restore_state` |

**Why:** Without locked defaults, every per-language guide hand-
waves "framec picks something idiomatic." Lock the table once, the
17 guides write themselves.

### D3 — Erlang fit (resolved by spike, see Phase 1.0)

**Decision (pending spike result):** Erlang's `gen_statem:start_link/N`
is the canonical construction call. Framec emits the user's factory
as the `start_link/N` clause. The blank-allocator equivalent is
`gen_statem:start_link(?MODULE, [], [{restore, true}])` — passing
a `restore` flag in the StartArgs that `init/1` reads to skip user
construction work.

The spike (Phase 1.0) verifies whether existing patterns cover this
or whether framec needs a small Erlang codegen extension to thread
the restore flag through `init/1`. Memory entry "Erlang persist ×
state-args" indicates `frame_state_args` and `frame_enter_args`
already round-trip via the gen_statem Data record; the blank-init
path may need only a single new `case StartArgs of [{restore,
true}] -> ...` branch in the generated `init/1`.

If the spike surfaces architectural friction, this RFC is updated
before backend implementation begins.

### D4 — Restore must skip start-state init logic

**Decision:** Restore allocates a blank instance and calls
`@@[load]`. It does NOT fire the start state's `$Start(...)` body
or `$>(...)` handler. The compartment is restored byte-equivalent
to the saved one; user code does not re-run.

This is the entire point of the factory-only model. To prevent
regression, **a dedicated matrix test (Test 91) is added in Phase 5**
that:

1. Constructs an instance whose `$>(seed: int)` handler increments
   a `domain.fire_count` field.
2. Saves the instance.
3. Restores into a fresh instance.
4. Asserts `fire_count` is the saved value (not incremented again).

Test 91 runs across all 17 backends; any divergence is a hard
blocker.

### D5 — Zero-arg factory case

**Decision:** Systems with no `@@system`-level params auto-generate
a zero-arg factory:

| Backend | Zero-arg factory rendering |
|---|---|
| Python | `Trivial()` |
| JavaScript / TypeScript | `new Trivial()` |
| Java / C# / Kotlin / Swift | `new Trivial()` (public no-arg ctor) |
| Rust | `Trivial::new()` |
| GDScript | `Trivial.create()` (static) |
| Go | `NewTrivial()` |
| C | `Trivial_create()` |
| C++ | `Trivial::create()` |
| Dart | `Trivial.create()` |
| PHP | `new Trivial()` |
| Ruby | `Trivial.new` |
| Lua | `Trivial.create()` |
| Erlang | `trivial:start_link()` (per D3) |

These are the same shapes as D2 with empty argument lists. No
special case needed in the design.

### D6 — Composition restore

**Decision:** When a system declares another `@@system` as a
domain field with constructor args:

```frame
@@system Outer {
    domain:
        inner = @@Inner(42)
}
```

Restore of `Outer` does NOT call `@@Inner(42)` again to reconstruct
`inner`. Instead, restore:

1. Calls `Inner`'s blank allocator (zero-arg).
2. Recursively `@@[load]`s the saved nested bytes into the blank
   `Inner` instance.

This is the explicit fix for FRAMEC_BUGS.md Issue #2. The
codegen path that previously emitted `self.inner = Inner.new()`
(zero-arg call to user `_init` with required params, crashing)
now emits `self.inner = Inner.<blank_allocator>()` followed by
`self.inner.<load_op>(saved_bytes)`.

The blank allocator name is the framec-internal name (always
zero-arg, always empty body), distinct from the user-facing
factory. Per-backend internal name convention: `__frame_alloc`
or `_blank_alloc` — finalized during Phase 1.1 (Python first).

---

## Note: rejected design

**`@@[destroy]` is not part of this RFC.** Destruction semantics
are non-uniform across hosts (6 of 17 backends have reliable
destructor hooks; 11 don't) and Frame's value proposition is
determinism. Users who need teardown logic write it as a regular
event handler or operation and invoke it explicitly. The state
machine model is sufficient.

## Implementation plan

### Phase 1 — Codegen migration (per-backend)

For each of the 17 backends:

1. Modify the system-level codegen to emit empty blank allocator
   instead of user-code-bearing constructor.
2. Modify domain-default emission to call the auto-generated
   factory with `@@system` params, instead of relying on the
   constructor.
3. Modify the persist restore path to call the blank allocator,
   not the user factory.
4. Run matrix tests; verify no regression.

This is mechanical work, parallelisable across backends.

### Phase 2 — Attribute scanner

Extend the RFC-0013 attribute scanner to recognize
`@@[create("name")]` at system level. Validate: single string
argument, no body permitted.

### Phase 3 — Validator

New error codes (system-level attribute, name-only):

- **E815** — `@@[create]`, `@@[save]`, `@@[load]` are system-level
  attributes, not operation attributes. Declaring them inside
  `operations:` is rejected.
- **E816** — each lifecycle attribute takes at most one argument:
  the desired user-facing name. Other forms (multi-arg, body, etc.)
  are rejected.
- **E817** — name argument, when present, must be a valid identifier
  in the host language. Framec rejects reserved words per backend.
- **E818** — at most one `@@[create]`, one `@@[save]`, one
  `@@[load]` per system.
- **E819** — RFC-0012 operation-attribute form (`operations:
  @@[save] foo(): T`) is rejected. Error message includes a
  pointer to the codemod (`scripts/migrate_rfc0015.py`) and an
  example of the new shape.

This removes RFC-0012's E811-E813 (operation-attribute form for
save/load). Migration: codemod ships with 4.1.0 release; see
"RFC-0012 supersession" below for details.

### Phase 4 — Cookbook + per-language guides

Update each per-language guide with the factory pattern and
`@@[create]` rename example. Add a cookbook recipe (around #34
range) that demonstrates init-logic-in-`$>`.

### Phase 5 — Fuzz coverage

Add a new fuzz phase or extend Phase 7 (multi-system) with
parameterized sub-systems × persist round-trip to lock in the
Issue #2 fix.

### Phase 6 — Migration codemod

A small Python script that scans existing Frame fixtures for any
init-time logic in non-`$>` locations (rare) and refactors them.
The cookbook + matrix corpus has been audited; no real cases need
the codemod, so it's a safety net for community fixtures.
