# RFC-0016: `@@[persist(domain=[...])]` — Selective Domain Persist

**Status:** Draft (2026-05-04). Deferred from 4.0 / 4.1.0 release scope.
**Created:** 2026-05-04
**Builds on:** RFC-0012 (`@@[persist]` + `@@[save]`/`@@[load]`),
RFC-0015 (factory-only construction).

---

## Motivation

The current persist contract serializes the **entire** state of a
`@@[persist(<type>)]` system: current state, state stack, all
state-vars and state-args, every domain field. For most state
machines that's the right default — they fit in a snapshot, the
state machine *is* the data.

For larger machines, three real classes of fields don't belong in
the snapshot:

1. **Derived fields.** A cache or memoization slot that's a pure
   function of other fields. Persisting it doubles the blob size for
   no information gain.
2. **Resource handles.** An open socket, file handle, database
   connection, or thread pool. Serializing it produces a value that
   is meaningless across the restore boundary; the receiver must
   re-open from scratch anyway.
3. **Cross-instance shared state.** A reference to another live
   `@@system` that the parent doesn't own (passed in at
   construction). Restoring the snapshot in a fresh process won't
   know how to wire that reference back; the user always handles it
   externally.

The cookbook recipe 107 (`MissionController`) used a syntax that
hinted at this feature in its `@@[persist(domain=["mission_id",
"site_list", "site_index", "findings", "inspection_retries"])]`
attribute — explicitly listing the domain fields to include. The
example didn't compile against the current contract; it was lifted
from an earlier design sketch. Recipe 107 has been migrated to the
RFC-0015-canonical form (`@@[persist(<type>)]` + `@@[save(name)]` +
`@@[load(name)]`) for 4.0; this RFC captures the pending design.

---

## Proposed syntax

A second attribute, distinct from `@@[persist(<type>)]`, declares
the field selector:

```frame
@@[persist(bytes)]
@@[save(save_state)]
@@[load(restore_state)]
@@[persist_fields(["mission_id", "site_list", "findings"])]
@@system MissionController { ... }
```

Two attributes, two responsibilities:

- `@@[persist(<type>)]` — what blob type the save method returns
  and the load method consumes. Mandatory for any persisted system.
- `@@[persist_fields([...])]` — optional. Lists the domain fields
  that round-trip through save/restore. If absent, every domain
  field round-trips (the current default).

The `@@[no_persist]` per-field attribute (already accepted by the
attribute scanner per RFC-0012) is the inverse — opt **out** of
persist on a per-field basis. Either form is valid; the
`@@[persist_fields]` shape is preferred when the included list is
short, `@@[no_persist]` when the excluded list is short.

Rejected variant: collapsing the selector into the type attribute
itself (`@@[persist(domain=[...])]`). That overloads `@@[persist]`'s
arg with two unrelated concerns (blob type + field selector) and
breaks the "one attribute, one responsibility" principle that
RFC-0015 deliberately established.

---

## Semantics

On save:
- Iterate the domain fields. For each:
  - If `@@[no_persist]` — skip.
  - Else if `@@[persist_fields([...])]` is declared and the field
    is **not** in the list — skip.
  - Else — serialize.
- Always include: current state, state stack, state-vars, state-args.

On load:
- Restore the always-included parts (state, stack, state-vars,
  state-args) from the blob.
- For each domain field:
  - If serialized — deserialize.
  - If skipped — leave at the default value the blank allocator
    produced (RFC-0015's contract).

Both `@@[persist_fields]` and `@@[no_persist]` are static — they
don't read the runtime state. Serialization is type-driven,
matching today's behavior.

---

## Validation

New error codes:

- **E820** — `@@[persist_fields]` is a system-level attribute, not
  per-field. Reject if attached to a domain field.
- **E821** — `@@[persist_fields]` requires the system to also have
  `@@[persist]`. Bare `@@[persist_fields]` without persist is
  rejected.
- **E822** — `@@[persist_fields]` argument must be an array of
  identifiers. Each identifier must name a domain field of the
  system (typo'd or non-existent fields rejected).
- **E823** — a single field cannot have both `@@[no_persist]` (per-
  field opt-out) and appear in `@@[persist_fields]` (per-system
  inclusion). Pick one or the other.

---

## Migration

The feature is purely additive:

- Existing `@@[persist(<type>)]` systems continue to round-trip
  every domain field (default).
- Adding `@@[persist_fields(...)]` to an existing system narrows
  the round-trip set; restored fields outside the list reset to
  their default values. Authors opt in.
- The 4.0 / 4.1.0 release ships **without** this attribute. Users
  who need partial-persist before this RFC lands have two options:
  - Mark non-persistable fields with `@@[no_persist]` (RFC-0012)
  - Move resource-handle / cache fields into a sibling system that
    isn't persisted

---

## Why deferred from 4.0

Three reasons:

1. **Real-world demand isn't proven.** Cookbook recipe 107 was the
   only place the feature was sketched; it didn't ship working in
   any 4.0.x release. No customer has requested partial-persist
   selectivity since RFC-0012 landed.
2. **`@@[no_persist]` covers the common case.** Per-field opt-out
   (RFC-0012's already-accepted attribute) handles "skip this one
   resource handle" cleanly. The selector form is only better when
   most fields would need `@@[no_persist]`.
3. **One persist redesign per release.** RFC-0015 is already a
   structural change to persist (factory-only construction, hard-cut
   from RFC-0012's op-attribute form). Stacking another semantic
   change on the same release creates avoidable migration risk.

This RFC is the place to track the design once it's ready to land.
Reopen the file when a real customer use case shows up.
