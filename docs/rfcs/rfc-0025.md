if/then/else asymmetries listed in your bug log at the grammar
level, not the codegen level.

The Oceans Model is an idea worth defending. Now make it a
calculus.

## Summary

Two parallel quality tracks against the framec codebase, plus one
deferred track that activates when scope expands:

- **Track A**: replace the 308 `.unwrap()` and 30 `panic!()` call
  sites with one of three intent-bearing forms:
  `.expect("invariant: …")`, structured `Result<_, CompileError>`
  propagation under a new error-code block (E900–E999, "internal
  invariant surfaced as recoverable error"), or `unreachable!()`
  with explanation.
- **Track B**: type the **Rust target's emitted dispatch
  infrastructure** with full Rust discipline. Replace the four
  `Box<dyn std::any::Any>` shapes that framec's Rust backend
  currently emits with three typed surfaces:
  - **Per-event enum** (`<System>Event`) for parameters — one
    variant per interface method, carrying that method's typed
    parameter list as named fields.
  - **Per-system return enum** (`<System>Return`) for handler
    return values — one variant per handler-with-return,
    carrying that handler's declared return type.
  - **Value enum** (`<System>Value`) for the `@@:data` map
    *only* — the one place runtime variance is genuinely
    needed (dynamic keys).

  Drop the `downcast-rs` dependency (already unused). Drop the
  manual `Clone` impl that silently dropped parameters. Drop the
  `unreachable!()` lies that today's `.downcast_ref::<T>()`
  pattern encodes. *Postmortem note: an earlier draft scoped this
  track to a refactor of framec's internal `dyn Any` payloads.
  That scope did not exist on disk — see Postmortem § below.*
- **Track C** (deferred): dedupe the `body_closer/` +
  `native_region_scanner/` per-language glue (17 files × ~500 LOC
  each of near-identical trait dispatch). **Not part of immediate
  execution.** Triggers automatically when an 18th backend is
  added or when the shared trait surface changes.

Tracks A and B ship now. Neither changes the user-facing Frame
language semantics or any system's public method surface. Track A
turns malformed-input panics into typed diagnostics. Track B
changes the *internal shape* of generated Rust target code:
anyone using a Frame system the intended way (edit `.frm`,
regenerate, call interface methods) sees cleaner generated code
on their next regen and no behavioral change. Track C is
documented here so the trigger is discoverable from the RFC, not
buried in narrative.

## Motivation

The framec CLI is shipping to users who paste arbitrary `.frm`
input. Every `.unwrap()` is a possible silent assumption violated
by a user input that nobody anticipated, surfacing as an opaque
backtrace. The error-code system (E400-series, W700-series) is
already in place and good; the panicky paths are bypassing it.

The `dyn Any` payload in the persistence codegen specifically
subverts the type discipline that justifies writing the rest of
framec in Rust. It exists because at codegen time the
state-arg/enter-arg/domain-field value space is "Frame's typed
value set," and the implementer reached for `Any` instead of
encoding that set in Rust types. The set is small (≤6 base shapes
once `List`/`Dict` recursion is included). It can be an enum.

Both fixes are localized to well-bounded surface area, support
incremental rollout, and require no migration on the part of Frame
authors.

## Track A — Unwrap / panic sweep → structured errors

### Baseline

```bash
rg '\.unwrap\(\)'   framec/src/ | wc -l   # ~308 expected
rg 'panic!\('       framec/src/ | wc -l   # ~30 expected
rg 'todo!\('        framec/src/ | wc -l   # 0
rg 'unimplemented!' framec/src/ | wc -l   # 0
```

### Per-site triage rule

For each `.unwrap()` / `panic!()` call site:

1. **Provable invariant**: the prior code path guarantees the
   `Some` / `Ok` case. Convert to
   `.expect("invariant: <what> guaranteed by <where>")`. The text
   names the upstream guarantor so a future reader can verify.
2. **Recoverable error**: the failure case represents a malformed
   input or an unsupported feature. Convert the function signature
   to `-> Result<_, CompileError>` and emit a structured error from
   the new E900–E999 block, "internal invariant surfaced as
   recoverable error." Use `?` to propagate. The error message
   describes what the caller did and what to do about it.
3. **Truly unreachable**: dead arms of exhaustive matches, etc.
   Convert to `unreachable!("reason")` with a one-sentence
   explanation.

### Wave plan

Three waves, each independently shippable, sequenced by blast
radius:

- **Wave 1 — `framec/src/frame_c/compiler/pipeline/`**: top-level
  compile orchestration. Highest user-facing surface. Estimated
  60–80 call sites.
- **Wave 2 — `framec/src/frame_c/compiler/codegen/backends/` + the
  17 `<lang>_system/` modules**: large surface, mechanical.
  Estimated 180–220 call sites.
- **Wave 3 — `framec/src/frame_c/compiler/{lexer,parser,arcanum}/`**:
  internal, lower user-visibility. Estimated 50–70 call sites.

### Error code block

Allocate **E900–E999**: "Internal invariant surfaced as recoverable
error." This range is for cases where framec's internal model
asserted a property that the input violated — e.g. "expected
non-empty state list at this point in the pipeline."

The first few codes are minted in Wave 1 as needed; they're not
pre-allocated.

### Verification

After each wave:

- `cargo test` shows 400/400 passing.
- `framec-test-env` matrix runs clean (no new failures introduced).
- Smoke test: 5 hand-crafted malformed `.frm` inputs (truncated
  systems, mismatched braces, undeclared symbols, deeply nested
  type errors, encoding garbage) — each produces a typed
  `CompileError` with a code, not a Rust backtrace.

At the end of Wave 3:

```bash
rg '\.unwrap\(\)' framec/src/ | wc -l    # target: under 50
rg 'panic!\('     framec/src/ | wc -l    # target: 0 outside test code
```

## Postmortem on the original Track B target

The first draft of this RFC scoped Track B to a refactor of
framec's *internal* `dyn Any` use, citing
`rust_system/persistence.rs` with 7 sites. **That file does not
exist on disk.** Verification:

```bash
$ find framec/src -name "persistence.rs"
# (no output)
$ rg "dyn Any" framec/src/
# 8 mentions, all in framec's Rust backend or in .gen.rs
# metatranspiler artifacts — every one is about CODE EMITTED into
# generated Rust target output, not framec's internal types
$ rg "use downcast_rs" framec/src/
# (no output — the downcast-rs Cargo dep is unused)
```

The mistake came from the persona-critique survey conflating two
distinct things:

1. **framec emits `Box<dyn Any>`** in the Rust target code it
   generates (necessary at the time because Rust has no native
   variant type and the Rust backend mirrored the dynamic-language
   backends' message-string dispatch shape).
2. **framec uses `Box<dyn Any>` internally** — *which it does
   not.* No internal payloads are `Box<dyn Any>`.

The reframed Track B below addresses (1) — the real opportunity,
larger in user impact and more aligned with the original "stop
throwing away Rust's type system" intent. The `downcast-rs` dep
removal still applies and is folded into the verification
checklist.

## Track B — Type the Rust target's emitted dispatch infrastructure

### Scope

Four distinct `Box<dyn std::any::Any>` shapes are emitted into
generated Rust output by `framec/src/frame_c/compiler/codegen/runtime.rs`
and `framec/src/frame_c/compiler/codegen/rust_system.rs`:

1. `Vec<Box<dyn Any>>` — event parameters, heterogeneous-by-position,
   known at Frame-compile time from interface declarations.
2. `Option<Box<dyn Any>>` — handler `_return` value.
3. `HashMap<String, Box<dyn Any>>` — `@@:data[key]` map,
   genuinely heterogeneous-by-key.
4. `Box<dyn Any>` — default-return slot, same shape as (2).

Each call site uses `.downcast_ref::<T>()` to read, with
`.cloned().unwrap_or_default()` as the panic-safety net. The
downcasts are type-correct by construction (Frame's type system
knows the type at codegen time), but the Rust type system can't
see that — every read carries a runtime cost and a silent
failure mode.

### Type discipline (Option C — full purist)

The design rejects two cheaper-but-leaky alternatives the team
explicitly considered and rejected during scoping:

- **Value-enum-everywhere** ("Option A"): use one closed enum
  (`Int`, `Float`, `Bool`, `Str`, `List`, `Dict`) for all three
  Box-shapes. Eliminates `dyn Any` but the read sites still
  pattern-match on the value enum with `unreachable!()` fallbacks
  — same type-erasure smell, just with a closed enum instead of
  an open one. Ships in 2 days. **Rejected**: encodes statically-
  known information at runtime.
- **Partial-typing** ("Option B"): per-event enum for parameters,
  value enum for `_data`, leave `_return` as `Box<dyn Any>`.
  Ships in 2.5 days. **Rejected**: leaves a third of the smell.

Option C is the full-purist target. Three typed surfaces, each
one expressing static information statically:

```rust
// (1) Per-event enum: one variant per interface method, carrying
//     that method's typed parameter list as named fields.
//     Replaces parameters: Vec<Box<dyn Any>>.
//
//     Derives Clone trivially (every field type implements Clone
//     by Frame's type contract).
#[derive(Clone, Debug)]
enum FooEvent {
    Connect { url: String, port: i32 },
    Tick,
    GetStatus,
    Shutdown { reason: String },
}

// (2)+(4) Per-system return enum: one variant per
//     handler-with-return, carrying the handler's declared
//     return type. Replaces _return: Option<Box<dyn Any>> on
//     FrameContext.
//
//     Handlers with void return don't get a variant; the enum
//     only carries the value-returning handlers' return types.
#[derive(Clone, Debug)]
enum FooReturn {
    GetStatus(String),
    GetCount(i64),
    // void handlers (Connect, Tick, Shutdown) don't appear here
}

// (3) Per-system value enum for the @@:data map. This is the
//     only place runtime variance is genuinely needed — keys
//     are dynamic, types are determined by the value pushed in.
//     Closed enum is still strictly better than dyn Any.
#[derive(Clone, Debug)]
enum FooValue {
    Int(i64),
    Float(f64),
    Bool(bool),
    Str(String),
    List(Vec<FooValue>),
    Dict(std::collections::HashMap<String, FooValue>),
}
// type FooData = HashMap<String, FooValue>;
```

After the change, a handler dispatch site goes from this:

```rust
// before
fn _state_Idle(&mut self, __e: &FooFrameEvent) {
    match __e.message.as_str() {
        "connect" => {
            let __ctx_event = &self._context_stack.last().unwrap().event;
            let url: String = __ctx_event.parameters.get(0)
                .and_then(|v| v.downcast_ref::<String>())
                .cloned().unwrap_or_default();
            let port: i32 = __ctx_event.parameters.get(1)
                .and_then(|v| v.downcast_ref::<String>())
                .and_then(|s| s.parse().ok())
                .unwrap_or_default();
            self._s_Idle_hdl_connect(__e, url, port);
        }
        _ => {}
    }
}
```

to this:

```rust
// after
fn _state_Idle(&mut self, __e: &FooEvent) {
    match __e {
        FooEvent::Connect { url, port } => {
            self._s_Idle_hdl_connect(__e, url.clone(), *port)
        }
        _ => {}  // statically-unreachable arms (compiler proves it
                 //  when match is exhaustive over the variants this
                 //  state handles)
    }
}
```

The `@@:return` read site goes from:

```rust
// before — read via downcast on Option<Box<dyn Any>>
let result: String = self._context_stack.last().unwrap()._return
    .as_ref()
    .and_then(|v| v.downcast_ref::<String>())
    .cloned().unwrap_or_default();
```

to:

```rust
// after — read via pattern match on the typed return enum.
//   framec knows which handler produced this return at codegen
//   time, so it emits the exact variant arm. The _ arm is the
//   only `unreachable!()` and it's a true compile-time
//   impossibility (a different variant would mean dispatch was
//   routed wrong, which framec's emission rules out).
let result: String = match self._context_stack.last().unwrap()._return {
    Some(FooReturn::GetStatus(ref s)) => s.clone(),
    _ => unreachable!("dispatch invariant: get_status returns GetStatus"),
};
```

Note: the `unreachable!()` survives only at the `_return` read
site, and it's a *true compile-time impossibility* — framec
emits this read after a specific handler call, so the variant is
statically known. Compare to the value-enum-everywhere approach
where `unreachable!()` would be at every parameter read site,
many per handler, justified only by Frame source semantics.

### Replacement steps

**B.1 — Per-event enum for parameters** (~1.5 days)

1. Add per-event enum emission to `rust_system.rs`. One variant
   per interface method (`SystemAst::interface_methods()`),
   named fields matching parameter declarations. Derives:
   `Clone`, `Debug`.
2. Replace the `FrameEvent.parameters: Vec<Box<dyn Any>>` field
   with the event enum. The `message: String` field collapses —
   the enum variant *is* the message identifier.
3. Rewrite per-state dispatch (`_state_X` functions) to match on
   the enum variant + destructure parameters directly. Pattern:
   ```rust
   fn _state_Idle(&mut self, __e: &FooEvent) {
       match __e {
           FooEvent::Connect { url, port } => {
               self._s_Idle_hdl_connect(__e, url.clone(), *port)
           }
           _ => {}
       }
   }
   ```
4. The manual `Clone` impl that dropped parameters goes away —
   `#[derive(Clone)]` on the enum is total (every variant field
   implements Clone by Frame's type contract).

**B.2 — Per-system return enum for `_return`** (~1 day)

5. Inventory handlers-with-return via `SystemAst::interface_methods()`
   filtered by non-void return declaration.
6. Emit a per-system return enum: one variant per
   handler-with-return, carrying that handler's declared return
   type. Variant naming: PascalCase of the interface method
   name (`FooReturn::GetStatus(String)`).
7. Change `FrameContext._return` field type from
   `Option<Box<dyn Any>>` to `Option<<System>Return>`.
8. Handler bodies that today write `self._return = Some(Box::new(value))`
   change to `self._return = Some(<System>Return::HandlerName(value))`.
   This happens in `frame_expansion/context_return.rs`'s
   Rust-target branch.
9. `@@:return` read sites pattern-match on the variant. framec
   already tracks the producing handler at every `@@:return`
   site — emit the exact variant arm; the `_` fallback is true
   compile-time-unreachable.

**B.3 — Value enum for `_data`** (~0.5 day)

10. Emit per-system `<System>Value` enum with 6 variants:
    `Int(i64)`, `Float(f64)`, `Bool(bool)`, `Str(String)`,
    `List(Vec<Self>)`, `Dict(HashMap<String, Self>)`.
11. Replace `_data: HashMap<String, Box<dyn Any>>` with
    `_data: HashMap<String, <System>Value>`. The `@@:data[key]`
    read sites (in `frame_expansion/handler_body.rs`'s
    Rust-target branch) pattern-match on `<System>Value`.

**B.4 — Bootstrap regen + cleanup** (~0.5 day)

12. Regenerate the 5 `.gen.rs` metatranspiler files via `framec
    compile -l rust`. These bootstrap files used the old
    `Box<dyn Any>` shape because they were emitted by the OLD
    Rust backend; after B.1–B.3 they regenerate with the new
    shape.
13. Drop `downcast-rs = "1.2.0"` from `framec/Cargo.toml`.
    Already unused (`rg "use downcast_rs" framec/src/` is
    empty); after Track B nothing references it.

**Total: ~3.5–4 days. Each phase shippable independently** —
B.1 alone eliminates the most-used `Box<dyn Any>` surface
(parameters, touched by every handler); B.2 and B.3 chase the
remaining two; B.4 cleans bootstrap.

### Validation step — kernel-parameterization borrow-checker risk

Before code change, verify the current Rust kernel's shape. The
original `Box<dyn Any>` design may have been load-bearing for a
borrow-checker problem in an earlier kernel iteration where the
kernel signature did not take an event-typed parameter. If the
current kernel still has that constraint, the typed-enum
rewrite has to thread the same needle (typically: borrowing
the event vs owning the parameters, mutability of the
context-stack, lifetime of the parameters vector across
transitions).

Concrete validation tasks (perform **before** starting step 1):

- Read `framec/src/frame_c/compiler/codegen/runtime.rs` kernel
  generation and identify the current signature of the
  per-state `__kernel` / `__router` / handler dispatch methods.
  Confirm whether the event is currently passed by value, by
  reference, by `Rc`, or by ownership-transfer through the
  context stack.
- Inspect one representative `.gen.rs` (e.g.,
  `attribute_scanner.gen.rs`) and trace how an event flows from
  external entry through the context stack into a handler.
  Confirm the lifetime constraints.
- Compile a minimal typed-enum sketch in a scratch file
  (`/tmp/typed_event_sketch.rs`) that mirrors the proposed
  shape; verify `cargo check` passes with the same dispatch
  topology framec currently emits. If the borrow checker
  rejects the sketch, the kernel shape needs adjustment in
  step 3 — record the adjustment in the RFC before proceeding.
- Git-log the introduction of `Box<dyn Any>` in `runtime.rs`
  (`git log -p -- framec/src/frame_c/compiler/codegen/runtime.rs
  | rg -B5 'Box<dyn'` or commit-range bisect) to recover the
  original rationale. If a commit message says "switched to
  Box<dyn Any> to satisfy borrow checker on kernel X," the
  rewrite must replay that fix in typed form.

### Verification

After each sub-phase:

- `cargo test` 400/400 (framec's own tests).
- All `.frs` fixtures across `framec-test-env` matrix run
  clean. Per-phase: B.1 should not affect persist or `@@:data`
  fixtures (they touch different surfaces); B.2 affects
  `@@:return` fixtures; B.3 affects `@@:data` fixtures.

After full Track B:

- `rg "Box<dyn" framec/src/frame_c/compiler/codegen/runtime.rs`
  returns nothing.
- `rg "Box<dyn" framec/src/frame_c/compiler/codegen/rust_system.rs`
  returns nothing.
- `rg "Box<dyn" framec/src/frame_c/compiler/attribute_scanner/*.gen.rs
  framec/src/frame_c/compiler/gdscript_multisys/*.gen.rs`
  returns nothing (bootstrap regen verified).
- `rg "downcast-rs" framec/Cargo.toml` returns nothing.
- `rg "manual Clone|doesn't implement Clone" framec/src/`
  returns nothing (the manual-Clone-with-dropped-parameters
  comment is gone — `#[derive(Clone)]` carries the load).
- Hand-spot-check: pick a fixture with `@@:return` access,
  inspect the generated Rust, confirm the read site is a typed
  variant match on `<System>Return`. Pick another with
  `@@:data[key]` access, confirm `<System>Value` pattern match.
- Hand-spot-check: pick a fixture with a multi-parameter
  handler (`connect(url: String, port: i32)`), confirm dispatch
  is `match FooEvent::Connect { url, port } => ...` with no
  intermediate downcasts.
- RFC-0027 snapshot tests, if landed by then: re-bless. Diff
  the before/after `.snap` files in PR review to confirm the
  shape change is exactly what's intended (and not something
  accidental in some other backend).

## Track C — Body closer / native region scanner glue dedup (deferred)

**Status within this RFC: Deferred — track opens automatically
when scope changes.** No work scheduled today. This section exists
so the trigger condition, approach, and verification criteria are
captured in the RFC, not lost in narrative prose, so that whoever
hits the trigger doesn't have to re-derive the analysis.

### Scope

- `framec/src/frame_c/compiler/body_closer/` — 17 per-language
  glue files (`python.rs`, `ruby.rs`, `java.rs`, ...), each
  ~500 LOC of near-identical trait-dispatch boilerplate wrapping a
  language-specific generated FSM. Diffs across them are
  predominantly struct names (e.g., `BodyCloserPy` vs
  `BodyCloserRuby`) and the `.gen.rs` filename they `include!()`.
- `framec/src/frame_c/compiler/native_region_scanner/` — same
  17-file shape, same near-identical glue. Treat both modules
  together when the trigger fires; their refactor shares the
  pattern.

### Why deferred

Current cost is zero-friction:

- The 17 glue files are read-only-stable. Nobody touches them
  month-to-month.
- The duplication is invisible during normal codegen work.
- Refactoring now — macro extraction, regenerating all 17
  `.gen.rs` files via `framec compile -l rust`, validating
  round-trip across the matrix — costs more than it saves at the
  current scale.

In short: 17 files of identical glue is annoying in the abstract
but doesn't impede any current work. The work that would be saved
by refactoring isn't being attempted.

### Trigger conditions

**Primary trigger — adding an 18th backend.** Writing the 18th
glue file of identical boilerplate is the friction that justifies
the macro. The macro then pays for itself immediately on backend
#19. If/when an 18th backend is on the docket, this track activates
*before* the new backend is written, so the new backend uses the
deduped shape from the start.

**Secondary trigger — shared trait surface change.** Any change
to `body_closer/mod.rs`'s trait that forces 17 near-identical
edits across the glue files crosses the same friction threshold.
At that moment, the macro becomes the cheaper path even without a
new backend in flight.

### Approach when triggered

Two viable paths:

1. **`macro_rules!` extraction.** Define
   `body_closer_for!(LangIdent, "lang.gen.rs")` (and similarly for
   native_region_scanner) that expands to the per-language struct
   + trait impl. New backend = one macro invocation in
   `body_closer/mod.rs`. The `.gen.rs` files themselves remain
   unchanged; only the glue collapses.
2. **Extend the metatranspiler.** framec already generates parts
   of itself into `.gen.rs` files for FSM bodies. Extend the
   generator to emit the glue too, so each per-language file
   shrinks to `mod gen; pub use gen::*;`.

Option 1 is the pragmatic default — small, local, no impact on
the metatranspiler maintenance load. Option 2 is purer but
expands the metatranspiler's surface area, which has its own
bootstrap-fragility concerns (cf. the snarky-Reddit-dev review's
"chicken-and-egg" jab). Default to Option 1 unless the
metatranspiler is being reworked for another reason concurrently.

### Verification when triggered

- All 17 backends (or 18, post-addition) produce byte-identical
  generated output vs the pre-refactor baseline. If
  [RFC-0027](rfc-0027.md) snapshot tests have landed by then,
  this is `cargo test` with zero snapshot diffs (modulo
  intentional changes from the trigger work itself).
- `cargo test` passes whatever its count is at trigger time.
- `framec-test-env` matrix runs clean across all backends.
- `wc -l framec/src/frame_c/compiler/body_closer/*.rs` shows
  the expected reduction (target: each per-language file drops
  from ~500 LOC to <50 LOC; `mod.rs` grows to absorb the macro
  or the shared dispatch).

## Drawbacks

- **Track A** is mechanical but voluminous. ~300 call sites means
  roughly that many small code-review decisions. Sequencing into
  three waves limits the per-PR review surface; even so, the work
  is tedious.
- **Track B** changes the shape of generated Rust target code.
  Users who do the normal thing (`.frm` → regenerate → call
  interface methods) see no break — their next regen produces
  cleaner Rust. Users who *hand-wrote test code constructing
  `FooFrameEvent` directly*, or who *added their own methods to
  the generated impl that touch `parameters` / `_return` /
  `_data`*, hit the new enum shape on regen and update their
  code once. The added `FooValue` enum needs to grow as new
  Frame base types ship — "add an arm when you add a type" is
  a small ongoing tax in exchange for compile-time exhaustiveness.
- **Track B option-C cost.** Choosing the full-purist option C
  over the cheaper value-enum-everywhere (A) or partial-typing
  (B) buys static-information-stays-static at the cost of an
  extra ~1.5 days, primarily on the `_return` typing work.
  Justification: leaving any of the three Box-shapes
  type-erased puts back the same pattern (`unreachable!()` as
  "trust me, Frame's type system checked it") that motivated
  this RFC.
- **Track B's validation step is not optional.** The kernel
  signature is `fn __kernel(&mut self)` with the event acquired
  via clone-from-`Rc`-on-context-stack. The borrow-checker
  workaround for the missing event parameter is *already in
  place* (event is `Rc<FrameEvent>`, cloning is an O(1)
  refcount bump). The validation sketch in `/tmp/typed_event_sketch.rs`
  confirms the typed-enum shape compiles cleanly under this
  topology. The risk is not borrow-checker breakage but
  bootstrap-regen surprises in the 5 `.gen.rs` files —
  validation step #2 (trace `.gen.rs` event flow) catches
  these before the rewrite starts.
- **Track C** documents work that is *deliberately not happening*.
  The drawback is the same as the trigger condition: until the
  18th backend or a shared-trait change appears, the 17-file
  duplication keeps accumulating any small per-language drifts
  that sneak in. Mitigation: snapshot tests
  ([RFC-0027](rfc-0027.md)) would catch a per-backend glue drift
  during normal CI, so the duplication is bounded in practice even
  while it's bounded in principle.
- New error codes (E900–E999) add to the documentation surface;
  each one needs a one-line description in
  `docs/error_codes.md`.

## Unresolved questions

- **Where do `.expect()` messages live?** Inline string literal is
  simplest. Discussion: do we want a structured "invariant marker"
  type with a unique ID, so we could mine logs for the most-hit
  expectations? Probably overkill at framec's scale. Default: plain
  string.
- **Snapshot test pinning during Track A?** If [RFC-0027](rfc-0027.md)
  lands first, Track A's regressions surface as snapshot diffs
  instantly. Recommend RFC-0027 phase 1 lands before Track A wave 2
  starts.
- **`downcast-rs` removal**: if any other module uses it (besides
  the 7 sites under treatment), Track B doesn't remove the
  dependency. Quick `rg` decides at execution time.
