## Summary

Standardize Frame's domain section on the existing **`name : type = init`** form already used for state variables, interface parameters, and system parameters — eliminating the current "domain is strictly native code" exception. Add a single, opt-in multi-line escape hatch via parens around the initializer:

```
domain:
    count : int = 0
    label : str = "default"
    items : list = (
        [1, 2, 3, 4, 5]
    )
```

The form already works today (the `try_annotated_name` shape is tried first for every target — see `framec/src/frame_c/compiler/pipeline_parser/domain_native.rs:665`). This RFC promotes it to **the** canonical form, deprecates the per-language native shapes, and adds the `= ( … )` multi-line extension that does not exist today.

## Motivation / Problem

Frame's design promise — *one source spec, 17 target languages* — has one persistent leak: the `domain:` block. Every other declaration site already conforms to a single Frame syntax:

| Site | Form | Status |
|---|---|---|
| Interface params | `name: type` (type optional) | Frame syntax |
| Interface return + default | `(): type = default` | Frame syntax |
| State variables | `$.name : type = init` | Frame syntax |
| State / enter / exit args | `$(name: type)`, `$>(name: type)` | Frame syntax |
| System params | `@@system T(name: type, $(x: int))` | Frame syntax |
| **Domain** | **`name = init` (Python) or `int name = init` (C) — *native pass-through*** | **Outlier** |

Consequences of the outlier:

1. **Frame source is not portable across targets.** A `.fpy` domain block uses Python type-hint syntax; a `.fc` domain block uses C declaration syntax. Switching `@@target` requires rewriting domain.

2. **Documentation contradicts itself.** The README quickstart shows `name = "World"` in a `domain:` block. Readers reasonably conclude that's *Frame* syntax. It isn't — it's untyped Python, which only happens to be legal Python.

3. **Validator coverage stops at domain.** Frame can detect malformed handler bodies, undeclared states, type-omitted-on-static-target params (today emits warnings), but it cannot say anything about domain because it never parses the type. Errors surface late at the target compiler.

4. **Per-language shape parsers are dead weight long-term.** `domain_native.rs` already implements four parsers (`TypeFirst`, `AnnotatedName`, `GoStyle`, `BareName`) with a per-language priority dispatch (665ff). Each shape is a maintenance liability and a per-language fork in the user's mental model.

5. **Frame's "no type system" stance is not in tension with this change.** Types are already passed through as `Type::Custom(String)` opaque tokens for interface params, state vars, and system params. Domain types would work the same way — the user writes target-appropriate type names; Frame does not interpret them.

## Non-Goals

- **No type system.** Frame remains a state-machine DSL that does not interpret types. The type slot stays an opaque string. No type vocabulary, no generics grammar, no inference.
- **No initializer translation.** The initializer expression is still native code (or, with this RFC, native code optionally wrapped in `( … )`). Frame does not translate `[1, 2, 3]` to per-language list literals.
- **No portable type aliases.** A future RFC could introduce `int8`, `string`, `boolean` aliases that map per-target. This RFC does not.

## Design

### 1. Canonical Syntax

```
domain:
    <name> : <type> = <initializer>
```

| Slot | Required | Notes |
|---|---|---|
| `<name>` | yes | Identifier, valid in every target language |
| `:` | conditionally | Required when `<type>` is present. Omitted in the bare form. |
| `<type>` | optional for dynamic targets | Opaque token string. Frame does not parse it. Required for static targets (E605). |
| `=` | yes | Always required. A domain field with no initializer is rejected (E606). |
| `<initializer>` | yes | Native code; either single-line or `( … )` wrapped. |

**Bare form (dynamic targets only):**

```
domain:
    name = "World"           // Python, JS, Ruby, Lua, Erlang, GDScript, PHP
```

For static targets (C, C++, Java, C#, Rust, Go, Kotlin, Swift, Dart, TypeScript with strict mode), the type is required:

```
domain:
    count : int = 0
    name  : str = "World"
```

### 2. Multi-Line Initializer — `= ( … )`

When the initializer cannot fit on one line, the user wraps it in parens:

```
domain:
    items : list = (
        [
            1, 2, 3,
            4, 5, 6,
        ]
    )
    config : dict = (
        {
            "host": "x",
            "port": 8080,
        }
    )
```

**Wrapper-detection rule (precise):**

> The initializer is treated as paren-wrapped iff the **first non-whitespace token after `=`** is `(` AND its matching `)` is on a *later* line (i.e., not the same line as the `=`).

Consequences of this rule:

| Source | Wrapped? | Why |
|---|---|---|
| `x = (5)` | no | `(` and matching `)` on the same line |
| `x = (5 + 3)` | no | same |
| `x = f(1, 2)` | no | first token after `=` is `f`, not `(` |
| `x = (f(1, 2))` | no | `(` matches `)` on the same line |
| `x = (\n  5 + 3\n)` | yes | leading `(` does not close on the same line |
| `x = (5) + 3` | no | parens close on same line; `+ 3` is part of the expression |

When wrapped, Frame consumes from the leading `(` to the matching `)` as the initializer. The wrapping `( … )` is **stripped** from the captured initializer text — it does not appear in generated code unless the user wrote inner parens. Stripping avoids the Python-tuple gotcha (`x = (5)` is the int 5 in Python; `x = (5,)` is a 1-tuple — Frame does not change either).

### 3. Per-Language Type Slot

Frame does not validate types. The user writes target-appropriate type names:

```
// Python (.fpy)
count : int = 0
items : list = []
config : dict = {}

// Rust (.frs)
count : i32 = 0
items : Vec<i32> = vec![]
config : HashMap<String, i32> = HashMap::new()

// Java (.fjava)
count : int = 0
items : List<Integer> = new ArrayList<>()

// C (.fc)
count : int = 0
items : int* = NULL
```

This matches how interface params already work. There is no portable-type vocabulary in this RFC. A `.frm` file (target-agnostic source) is a future-RFC concern.

### 4. Type Slot Termination

The type slot ends at the first **top-level** `=`. "Top-level" means not nested inside `(`, `[`, `{`, or `<>` brackets, and not inside a string literal.

- `x : Map<K, V> = {}` — `<>` is part of the type
- `x : list[int] = []` — `[]` is part of the type
- `x : Tuple<int, int> = (0, 0)` — outer `<>` part of type, top-level `=` ends type slot

### 5. Tokenizer / Scanner Approach

The line scanner and the wrapper scanner share the same low-level tokenizer (already partly implemented in `domain_native.rs:tokenize`):

**Line scanner** (single-line decls):
1. Skip leading whitespace.
2. Tokenize the line into `Ident`, `Number`, `StringLit`, `Punct`. Skip trailing comments (per-target: `#`, `//`, `--`, `%`).
3. Match the canonical form: `Ident <name> Punct ":" <type-tokens> Punct "=" <init-text>` OR `Ident <name> Punct "=" <init-text>` (bare).
4. The type slot is captured token-by-token until a top-level `=` is found.
5. The init text is captured **verbatim** as the byte slice from the first byte after the top-level `=` to end-of-line (after stripping trailing whitespace and trailing comments).

**Wrapper scanner** (multi-line `= ( … )`):
1. After the top-level `=`, peek the first non-whitespace byte.
2. If `(` AND the line ends before the matching `)` is found:
   - Switch to bracket-aware scanning starting from the `(`.
   - Track depth across `(`/`)`, `[`/`]`, `{`/`}`.
   - Handle string literals using the per-target string-literal rules (delegate to the existing per-language skipper modules: `python_skipper.frs`, `c_skipper.frs`, etc.).
   - Handle comments inside the wrapper using the per-target comment rules (same skippers).
   - When depth returns to zero on a `)`, that's the closing wrapper. Capture the byte range *between* the wrapping `(` and `)` as the init text.
3. If the matching `)` is on the *same line*, treat as single-line (do not consume past EOL).

### 6. Domain Block Termination

Unchanged. The domain block ends at the next section keyword (`interface:`, `machine:`, `actions:`, `operations:`) or the closing `}` of the system. Empty lines and comment-only lines within the block are skipped.

### 7. Codegen

The captured `(name, type, init_text)` triple feeds the existing per-target backend emission (already implemented for the `AnnotatedName` shape). No new backend code is required — the RFC is parser-side. Per-target emission patterns (already in place):

| Target | Pattern | Example |
|---|---|---|
| Python | `self.<name> = <init>` (in `__init__`) | `self.count = 0` |
| TypeScript / JS | `this.<name> = <init>` | `this.count = 0` |
| Rust | field in struct + `<name>: <init>` in constructor | `Self { count: 0 }` |
| C | struct field + `frame->__domain.<name> = <init>` | `frame->__domain.count = 0` |
| C++ | member init list | `: count(0)` |
| Java / C# / Kotlin | field + `this.<name> = <init>` | `this.count = 0` |
| Go | struct field + `s.<Name> = <init>` (case-converted) | `s.Count = 0` |
| Swift | `self.<name> = <init>` | `self.count = 0` |
| PHP | `$this-><name> = <init>` | `$this->count = 0` |
| Ruby | `@<name> = <init>` | `@count = 0` |
| Lua | `self.<name> = <init>` | `self.count = 0` |
| Dart | `this.<name> = <init>` | `this.count = 0` |
| GDScript | `self.<name> = <init>` | `self.count = 0` |
| Erlang | record field | `#state{count = 0}` |

For the **type** slot, current behavior is retained: targets that need declarations (Rust struct fields, Java/C#/Kotlin fields, C struct members, etc.) emit `<name>: <type>` (or per-target equivalent) using the captured opaque type string.

### 8. Backwards Compatibility & Migration

The current parser already tries `AnnotatedName` first for every target — so today's typed domain decls (`count: int = 0` in any target) already work. **No working file is broken by *adding* RFC-0007's syntax.**

What this RFC *removes*:
- `TypeFirst` shape (`int count = 0`) — affects C, C++, Java, C#, Kotlin, Swift, Dart, Rust users
- `GoStyle` shape (`count int = 0`) — affects Go users
- `BareName` for static targets — affects users who omitted type on a static target

Migration tooling: a one-shot script (`scripts/migrate_domain_decls.py`) that scans `*.f*` files in a project and rewrites legacy shapes to canonical Frame form. Stretch goal: also detect and rewrite via AST traversal in framec itself (`framec migrate <file>`).

**Phased rollout:**

| Phase | Behavior | Diagnostic |
|---|---|---|
| **0** (current) | All four shapes accepted | none |
| **1** (this RFC, opt-in) | Canonical form recommended; legacy shapes accepted | W607 deprecation warning when legacy shape used |
| **2** (next minor) | Legacy shapes accepted only behind `@@allow_legacy_domain` directive | W607 promoted to E607 by default |
| **3** (next major) | Canonical form only; `@@allow_legacy_domain` ignored | E607 unconditional |

### 9. Compatibility With Other Frame Sites

The canonical form already aligns with state vars (`$. name : type = init`) and interface params (`name : type`). After this RFC, learners encounter **one** declaration form across the entire language.

## Error / Exception Conditions

New error and warning codes introduced:

| Code | Severity | Condition | Example |
|---|---|---|---|
| **E605** | error | Static target requires explicit type; bare form used | `domain: count = 0` in `.frs` (Rust) |
| **E606** | error | Domain field has no initializer | `count : int` (no `= …`) |
| **E607** | error / warning | Legacy domain shape used (Phase 2: warning; Phase 3: error) | `int count = 0` instead of `count : int = 0` |
| **E608** | error | Type slot is empty after `:` | `count : = 0` |
| **E609** | error | Multiple `=` operators not separated by brackets | `a = b = 0` (use `c : int = 0; b = c; a = b` or rework) |
| **E610** | error | Wrapper `(` opened but file ends before matching `)` | unterminated multi-line init |
| **E611** | error | Trailing tokens after wrapper `)` on its line | `items : list = (\n  [1,2]\n) + extra` — ambiguous |
| **E612** | error | Type slot contains unbalanced brackets | `x : Map<K, V = {}` (missing `>`) |
| **E613** | error | Domain field name shadows a system parameter | `@@system T(count: int) { domain: count : int = 0 }` (today: silently overridden) |
| **E614** | error | Domain field name duplicated within the block | two `count : int = 0` lines |
| **W615** | warning | Initializer references a name not in domain or system params | `count : int = unknown_helper()` |

**Comment-only and empty lines:** silently skipped (not an error).

**String literal containing `=` or `:` in initializer:** handled by the per-target string-literal awareness in the wrapper scanner. Single-line scanner does not need string awareness because it captures verbatim to EOL.

**Trailing comment on a single-line decl:** captured as part of EOL-to-end. Frame does not strip the comment from the captured init text — it survives into the generated code as a native comment. This is consistent with handler-body native pass-through.

**Wrapper `(` with content on the same line *before* a newline:**

```
items : list = (1, 2,
                3, 4)
```

The `(` does not close on the same line, so this is a wrapper. The captured initializer (after stripping outer parens) is:

```
1, 2,
                3, 4
```

The user is expected to write something the target language accepts when wrapped. Frame does not validate.

## Per-Language Compatibility Matrix

The canonical form `name : type = init` is **universally legal as captured**, but the *type slot text* and *init text* are still target-specific. The matrix below identifies which targets are "easy" (the canonical form maps directly to native syntax with minimal transformation) versus "needs special handling" (codegen does extra work).

### Group A — Direct fit (canonical form is also native syntax)

These languages natively use `name: type = init` declaration syntax. The domain block already looks like idiomatic native code in this form.

| Language | Notes |
|---|---|
| **Python** | Native PEP 526 type-hint form. `count: int = 0` is identical Frame and Python. |
| **TypeScript** | `count: number = 0` — same form. |
| **Rust** | `count: i32` (struct field syntax) — Frame emits the field; `= init` becomes the constructor expression. |
| **Swift** | `count: Int = 0` — direct map. |
| **Kotlin** | `count: Int = 0` (with `val`/`var` prepended in the emitted code). |
| **Dart** | `int count = 0` is Dart-native; `count: int = 0` is also accepted in Dart as a typedef-style field. Codegen emits the Dart-native `int count = 0`. |
| **GDScript** | `count: int = 0` — direct map. |

### Group B — Reorder needed (codegen rearranges to native syntax)

These languages declare in `type name = init` order. Codegen reorders the captured `(name, type, init)` triple at emission.

| Language | Native form | Codegen action |
|---|---|---|
| **C** | `int count = 0;` | Emit `<type> <name> = <init>;` in struct member or `__domain` init |
| **C++** | `int count = 0;` | Same as C; member init list for ctor |
| **Java** | `int count = 0;` | Field declaration order: type-first |
| **C#** | `int count = 0;` | Same |
| **Go** | `Count int` (no init in struct decl; init in constructor func) | Reorder + capitalize name (Go convention); init goes in `New<System>()` |

### Group C — Type optional / dynamic (type slot can be empty)

| Language | Notes |
|---|---|
| **JavaScript** | No types; `this.count = 0`. The bare form `count = 0` is sufficient. Frame still accepts the typed form (type ignored at emit). |
| **PHP** | Optional types since 7.4; bare or typed both fine. |
| **Ruby** | No types; `@count = 0`. Bare form. |
| **Lua** | No types; `self.count = 0`. Bare form. |
| **Erlang** | Record field; type captured but unused. Bare or typed both accepted. |

### Group D — Special considerations

| Language | Issue | Resolution |
|---|---|---|
| **Go** | Field name must be capitalized for export. | Codegen already case-converts (snake_case → PascalCase). No new work. |
| **Erlang** | Records are immutable; "field init" goes in the record default. | Codegen already handles. |
| **C** | No member init list; struct fields default to zero. Domain init runs in `<System>_init()`. | Existing pattern. |
| **Rust** | Lifetimes, references, and generics in the type slot can include `<>`. | Type-slot scanner already brackets-aware for `<>`. |
| **Java/Kotlin generics** | `List<Map<K,V>>` — nested `<>`. | Bracket scanner counts depth; covered. |
| **Swift / Kotlin / TS optionals** | `count : int? = 0` — `?` is a type-suffix character. | Add `?` to the type-token character set (one-line tokenizer change). |
| **Lua / Erlang `[]` literal** | Lua: `{}` not `[]`. Erlang: `[]` is a list. | Initializer is native; user writes target-correct. Frame does not translate. |

### Languages with no special handling required (covered by Groups A + B + C)

Python, TypeScript, JavaScript, Java, C#, Rust, Go, Kotlin, Swift, Dart, Ruby, PHP, Lua, GDScript, C, C++, Erlang — all 17 supported targets parse identically with the canonical form. Per-target divergence is *only* in the codegen emission, which is already implemented for the `AnnotatedName` shape today.

## Open Questions

1. **Multi-statement initializers.** Do we need `= ( … )` to allow a *block* (multi-statement, e.g., for languages where the field needs side-effecting init)? Current proposal: no — initializer is a single expression. Side-effects belong in an action or the constructor body.
2. **Inline trailing comments inside the wrapper.** Captured verbatim into the init text — okay for most targets (the comment passes into the generated code) but could leak per-target comment syntax. Acceptable; users opt into wrapping.
3. **Whether to require `:` even in the bare form** for visual symmetry (`count = 0` vs. `count : = 0`). Current proposal: no. Bare = no `:`.
4. **Whether `@@allow_legacy_domain` should ever exist** (Phase 2 escape hatch). Current proposal: yes — gives users one minor version to migrate. Removed in next major.

## Alternatives Considered

- **Status quo.** Domain remains native pass-through. Rejected — the README's silently-broken example proved the docs/code mismatch is a real user-visible problem.
- **Strict line-terminated, no multi-line escape (option A in the prior discussion).** Simpler parser, but real users will hit the line-length wall on dict/list literals. The wrapper escape is small and explicit.
- **Bracket-aware implicit continuation (option B).** No syntax addition, but invisibly-multi-line decls are debugging-hostile when the user misindents or unbalances brackets.
- **Backslash continuation (option C).** Alien to every target language.
- **Always-required braces (option D).** Ugly common case; double-wrap on dict/list literals.

The optional-paren approach is the minimum syntax addition that gives visible, unambiguous, all-target-friendly multi-line support without forcing it on the common case.
