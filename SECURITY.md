# Security Policy

## Reporting a Vulnerability

If you discover a security issue in framepiler — whether in the transpiler itself or in code patterns generated by a backend — please report it privately rather than opening a public issue.

**Preferred channel:** open a [GitHub security advisory](https://github.com/frame-lang/framepiler/security/advisories/new) on this repository. Advisories are private until coordinated disclosure.

**Alternate channel:** email `mark@frame-lang.org` with the subject prefix `[framepiler-security]`.

Please include:

- Affected version (`framec --version`)
- Target language(s) involved, if any
- Minimum reproducing Frame source
- Observed vs. expected behavior
- Any known mitigations or workarounds

## Scope

In scope:

- Bugs in the transpiler that allow malformed Frame source to crash the compiler in unsafe ways (e.g., out-of-bounds reads in unsafe code blocks)
- Code generation that emits target-language code with a security flaw not present in the Frame source (e.g., command injection, missing escaping)
- Validator gaps that allow Frame source to compile to code that violates documented Frame semantics in a security-relevant way

Out of scope:

- Vulnerabilities in user-written native code passed through unchanged by the transpiler
- Misuse of generated code outside its documented runtime contract
- Issues only affecting unsupported, experimental, or alpha-quality backends (see `docs/` for backend status)

## Response

Acknowledgement target: 5 business days. Initial assessment and remediation plan: 2 weeks.

Coordinated disclosure preferred. Once a fix is released, the advisory will be published with credit (unless you request otherwise).

## Supported Versions

Only the latest minor release of the `4.x` line currently receives security fixes. Earlier major versions are unsupported.
